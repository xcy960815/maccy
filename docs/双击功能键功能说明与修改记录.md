# 双击功能键功能说明与修改记录

## 功能概述

本次迭代为 Clipbook 增加了“通过双击修饰键唤醒”的打开方式。

当前功能形态如下：

- 入口位置：`Settings -> General -> Open`
- 默认状态：关闭
- 可选触发键：`Option`、`Shift`、`Control`
- 触发方式：快速双击同一个修饰键
- 触发行为：`toggle`，即如果窗口未打开则打开，如果窗口已打开则隐藏

## 当前交互规则

### 1. 设置页中的表现

当“双击唤醒”关闭时：

- 左侧显示原有的 `KeyboardShortcuts.Recorder`
- 用户继续使用普通快捷键，例如默认的 `⇧⌘C`

当“双击唤醒”开启时：

- 左侧改为“双击修饰键”选择框，显示样式与原快捷键录制框保持一致
- 开启开关时，会先清空当前双击键位选择
- 占位文案为“设置快捷键”
- 设置窗口会开始监听可用修饰键；用户双击 `Option`、`Shift` 或 `Control` 之一，即可完成选择
- 当已经选中某个键位时，输入框右侧保留清除图标，点击后可清空当前选择

### 2. 与原快捷键的关系

设计约束如下：

- 只保留一种打开方式生效
- 开启“双击唤醒”后，原有弹窗快捷键不会被删除，但会在运行时被禁用
- 关闭“双击唤醒”后，原有快捷键会自动恢复

这样做的原因是：

- 避免用户丢失自己之前配置过的快捷键
- 避免“双击唤醒”和“普通快捷键”同时生效造成行为重叠

### 3. 设置页面中的特殊处理

在设置窗口打开期间，唤醒行为应当失效：

- 普通弹窗快捷键不应在设置页中触发
- 双击修饰键也不应在设置页中触发

这样可以避免用户在录制或调整设置时误打开 Clipbook 主窗口。

## 代码实现说明

### 1. 配置项

本功能依赖两个设置项：

- `doubleClickPopupEnabled`
- `doubleClickModifierKey`

其中：

- `doubleClickPopupEnabled` 控制是否启用“双击唤醒”模式
- `doubleClickModifierKey` 记录当前选中的修饰键，取值为 `none`、`option`、`shift`、`control`

同时保留了旧数据迁移逻辑：

- 如果用户旧版本只存在 `doubleClickOptionKey`
- 升级后会自动迁移到新的通用双击修饰键配置

### 2. 检测逻辑

触发检测位于 `Popup` 相关逻辑中，核心思路如下：

- 监听 `flagsChanged` 和 `keyDown`
- 只识别“单独按下并释放的修饰键”
- 如果修饰键和其他键组成组合键，则本次按压不会被当作双击候选
- 两次独立点击间隔在阈值内时，判定为双击
- 仅允许 `Option`、`Shift`、`Control`
- 不允许 `Command`、`Fn` 等其他功能键触发

这保证了“只有明确支持的三个修饰键可用”，避免误触发。

### 3. 设置 UI

双击模式的录制行为仍然是项目内自定义的，但展示层最终改成了直接复用 `KeyboardShortcuts.RecorderCocoa` 作为底座，原因是：

- `KeyboardShortcuts.Recorder` 本身录制的是“普通快捷键”
- 本功能录制的是“允许双击的单个修饰键”
- 录制语义不能直接沿用三方库默认逻辑，但它的输入框样式、尺寸和清除图标布局是可以复用的

最终方案如下：

- 录制显示控件直接使用 `KeyboardShortcuts.RecorderCocoa`
- 输入框本身只负责展示当前选中的修饰键，不负责真正录制
- 真正的双击键位录制由 `DoubleClickModifierRecorder` 监听 `flagsChanged` / `keyDown` 完成
- 清除动作通过右侧透明点击层接管，点击原有清除图标区域时会把 `doubleClickModifierKey` 重置为 `.none`

这样做的好处是：

- 输入框高度和系统内其他快捷键输入框一致
- 圆角、边框、字体和清除图标都与原组件保持一致
- 既保留了自定义的“双击修饰键”录制语义，也避免了手写一套外观

## 本次协作处理记录

本次功能并不是一次性完成的，中间做过多轮修正和补强，整理如下。

### 第一阶段：从“仅支持双击 Option”扩展为“通用双击修饰键”

最初需求是“双击 Option 唤醒 Clipbook”。

后续调整为：

- 功能名从单一的“双击 Option”改为通用的“双击唤醒”
- 支持键位改为 `Option`、`Shift`、`Control`
- 默认关闭，由用户在设置中主动开启

### 第二阶段：限制触发范围，避免其他功能键误触发

中间重点确认过一个问题：

- 不能因为加入双击检测，就让 `Command`、`Fn` 等其他功能键也能唤醒 Clipbook

最终实现只允许：

- `Option`
- `Shift`
- `Control`

其他功能键不会触发。

### 第三阶段：只保留一种打开方式

根据产品决定，打开 Clipbook 的方式在同一时刻只保留一种有效模式：

- 开启双击模式时，普通快捷键运行时禁用
- 关闭双击模式时，普通快捷键恢复

注意：

- 这里不是清空原快捷键配置
- 而是在双击模式下临时禁用原快捷键

### 第四阶段：补充“重置设置”能力

设置页中增加了“重置设置”按钮，约束如下：

- 只重置设置
- 不清空剪贴板历史

按钮位置也做了调整，与“检查更新”按钮放在一起，便于发现。

### 第五阶段：修复设置页中的触发问题

后续又补了一条重要规则：

- 当用户位于设置页面时，打开 Clipbook 的触发方式不应生效

这包括：

- 普通快捷键
- 双击修饰键

避免在配置过程中发生误唤醒。

### 第六阶段：修复启动期回归问题

在一次实现中出现过启动问题，原因是：

- `Popup.init()` 期间过早读取了 `AppState.shared`
- 导致初始化链路互相依赖，应用无法正常启动

后续已将相关逻辑修正，避免在初始化早期触发该问题。

### 第七阶段：远程构建与发布链路调整

由于本地不安装 Xcode，本次还补了 GitHub Actions 的远程构建链路：

- 由 `tag` 触发构建
- 构建 unsigned 的 macOS app zip
- 将产物上传到 GitHub Releases

同时做了这些配套调整：

- 将工作流改为以 tag 发布为主，而不是普通 push 自动构建
- 修正 release 产物上传逻辑
- 增加 `appcast.xml` 的自动回写逻辑

### 第八阶段：修正检查更新地址

为了让应用中的“检查更新”使用当前 fork 仓库，而不是上游仓库，又补了以下修改：

- `SUFeedURL` 改为当前仓库地址
- `appcast.xml` 改为当前仓库 release 地址
- About 面板中的 GitHub / Support 链接改为当前仓库

### 第九阶段：补齐双击监听所需权限

双击修饰键要在应用外也能工作，单靠原来的辅助功能检查不够，因此又补了权限分层处理：

- `Accessibility.check(...)` 新增了 `listenEvent` 和 `postEvent` 参数
- 双击唤醒会主动检查 `CGPreflightListenEventAccess` / `CGRequestListenEventAccess`
- 粘贴模拟会主动检查 `CGPreflightPostEventAccess` / `CGRequestPostEventAccess`
- `Clipboard`、`PasteStack`、`Popup`、`GeneralSettingsPane` 都改成按需请求权限

这样可以减少“明明已经开了辅助功能，但双击修饰键仍然不生效”的问题。

### 第十阶段：兼容本地 Xcode 16.2 构建

为了让项目能在本地 Xcode 16.2 上继续编译，又补了兼容处理：

- `VisualEffectView` 中对 macOS 26 的玻璃效果视图做了降级
- 在 Xcode 16.2 下没有暴露的 `NSGlassEffectView`，统一退回到普通 `NSVisualEffectView`

这一步的目标不是改变 UI 设计，而是保证：

- 本地可以直接 `xcodebuild`
- 本地可以直接替换安装 `Clipbook.app`

### 第十一阶段：设置页样式与交互最终收口

这一阶段主要解决“双击模式输入框看起来不像原生快捷键输入框”的问题，最终定稿如下：

- placeholder 从“请双击 Option / Shift / Control”改为“设置快捷键”
- 开启双击模式时，先清空当前双击键位选择
- 去掉中途使用过的自定义外层容器
- 改为直接使用 `KeyboardShortcuts.RecorderCocoa` 本体显示当前键位
- 保留原组件的右侧清除图标区域，并将清除动作接回 `clearDoubleClickSelection()`
- 最终高度与“固定 / 删除 / 预览”等输入框保持一致

这一轮是设置页体验真正稳定下来的关键一步。

### 第十二阶段：补齐双击唤醒的权限兼容与本地调试说明

在后续本地联调中，又发现了一个更实际的问题：

- 用户已经在系统设置中打开了“输入监控”
- 设置页中的双击配置也已经开启
- 但双击 `Option` 仍然不能稳定唤醒 Clipbook

进一步排查后，问题分成了两类：

#### 1. 权限兼容策略过于严格

之前的实现默认把“双击修饰键唤醒”强绑定到 `ListenEvent` 权限路径：

- 如果没有 `Input Monitoring`
- 即使系统已经授予 `Accessibility`
- 双击模式也可能直接退回普通快捷键模式，导致用户看起来“已经开了权限，但功能不生效”

这次已将逻辑调整为：

- 优先使用 `ListenEvent` / event tap 监听全局修饰键
- 如果没有 `ListenEvent`，但已经有 `Accessibility`，则退回 `NSEvent.addGlobalMonitorForEvents` 作为兜底
- 只有当这两类能力都不可用时，双击模式才视为不可用

同时，设置页提示文案也同步改成了更准确的说明：

- 双击唤醒优先依赖 `Input Monitoring`
- 如果 `Input Monitoring` 不可用，则允许退回 `Accessibility` 兼容路径
- 不再把需求错误地描述成“只依赖监听输入事件权限”

#### 2. 本地未签名 / ad-hoc 构建的权限绑定问题

在本地用禁签名方式构建调试包时，还出现过另一类问题：

- 应用的 `CFBundleIdentifier` 仍然是 `com.xcy960815.Clipbook`
- 但实际代码签名标识可能变成了 `Clipbook` 或其他临时值
- macOS 的 TCC 权限记录绑定的是应用签名身份，而不是只看显示名称

这会导致一种很迷惑的现象：

- 系统设置里已经能看到 `Clipbook`
- “输入监控”或“辅助功能”似乎也已经打开
- 但对当前这份本地构建出来的 app 实际并不生效

因此，本地调试双击唤醒时还需要额外注意：

- 保证调试包的签名标识与 `com.xcy960815.Clipbook` 一致
- 如果中途替换过 app 包，必要时需要重置 TCC 记录并重新授权

## 当前已知限制

### 1. 构建产物仍是 unsigned

当前 GitHub Actions 构建出的产物为未签名版本，适合测试和分发，但如果后续要让 Sparkle 自动更新流程更完整，通常还需要进一步补齐：

- 应用签名
- Sparkle 相关签名信息

### 2. 双击唤醒依赖系统权限

如果缺少以下权限，双击修饰键可能无法正常工作：

- 监听输入事件权限
- 或辅助功能权限

当前实现会优先使用监听输入事件权限；如果该权限不可用，但辅助功能权限已经授予，则会退回兼容监听路径继续工作。

也就是说：

- 最理想的配置是开启“监听输入事件”
- 如果没有“监听输入事件”，只要“辅助功能”可用，双击唤醒仍有兼容路径
- 两种权限并不是都必须打开，至少一种可用即可
- 如果本地替换过未签名或重新签名的 app，仍可能需要重新授权

### 3. 双击录制逻辑仍是项目内自定义

最终展示控件已经改为复用 `KeyboardShortcuts.RecorderCocoa`，但真正的“双击修饰键识别”依旧由项目内的 `DoubleClickModifierKeyDetector` 与相关监听器自行维护。

这意味着后续如果要继续扩展：

- 新增支持的修饰键
- 调整双击时间阈值
- 增加更复杂的键位冲突处理

仍然需要优先修改项目自身逻辑，而不是只改三方库配置。

## 建议的后续维护方向

- 如果未来要支持更多唤醒方式，建议继续保持“同一时刻仅一种方式生效”的规则
- 如果未来要继续优化设置页体验，优先保持与 `KeyboardShortcuts.Recorder` 的视觉一致性
- 如果后续准备正式分发，建议补齐签名与 Sparkle 发布链路
